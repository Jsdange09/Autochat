Section 01

What Is This Project?

Aegis-Autochat is a self-healing AI chatbot system — one that doesn't just answer questions, but continuously watches its own performance, detects when it's giving bad answers, and automatically repairs its knowledge without anyone pressing a button.

Most AI chatbots are static. You train them, deploy them, and hope they stay relevant. When the world changes — when new documents arrive, when a topic wasn't covered — the bot silently gives wrong or vague answers. No alarm goes off. No one knows.

This project solves that problem by adding a complete autonomous feedback loop around a standard chatbot. The chatbot (built by one developer) is the brain that answers questions. The MLOps system (your part) is the nervous system that watches, detects, repairs, and deploys — all automatically.

Section 03

Every Tool — What It Does & Why

⚡
FastAPI
Serving Layer · API Gateway

What it is: A modern Python web framework for building REST APIs. Every HTTP request — from the React frontend, from your curl test, from Prometheus — goes through FastAPI.

What it does in this project: Hosts 5 endpoints. /chat is the main one — it receives a question, calls ChromaDB for context, calls Phi-3 for the answer, runs your confidence scoring, logs to Arize and W&B, and returns the result. All in one function.

Why FastAPI over Flask or Django? It's async-native, auto-generates API docs at /docs, has built-in Pydantic validation, and is 2–3× faster than Flask under load — ideal for a project that needs to prove performance.
🧠
llmware + bling-phi-3-gguf
The Brain · Local LLM

What it is: llmware is a Python library that manages local LLMs. bling-phi-3-gguf is Microsoft's Phi-3 model quantised into GGUF format — a compressed file format that lets large models run on a normal laptop CPU without a GPU.

What it does: Takes a prompt (question + context) and generates a natural language answer. It runs 100% locally — no internet, no API cost, no data leaving the machine.

Why local LLM over GPT-4 API? Privacy, zero API cost, offline capability, and proving to the panel that you understand model internals. A local LLM is harder to set up and more impressive than wrapping an API call.
🗄️
ChromaDB
Vector Store · Semantic Memory

What it is: A vector database — a database that stores numbers (embeddings) that represent the meaning of text, rather than the text itself.

What it does: When a question comes in, it's converted to a 384-dimensional vector. ChromaDB finds the stored vectors closest to it — those are the most semantically similar past answers or document chunks. This is what makes the chatbot "remember" things.

How it works: Text → SentenceTransformer → float[384] vector → stored. Query → same transform → cosine distance search → top-K results returned as context.

Why ChromaDB over Pinecone/Qdrant? Runs fully locally with zero setup, stores to disk automatically (backend/chroma_db/), and has a dead-simple Python API. Perfect for a project demo that needs to work reliably without cloud services.
🔢
SentenceTransformers
Embedding Engine · Shared by Both Parts

What it is: A Python library wrapping pre-trained transformer models that convert text into dense numerical vectors that capture semantic meaning.

Model used: all-MiniLM-L6-v2 — a 22MB model that produces 384-dimensional embeddings. It's fast, accurate, and runs on CPU.

Dual use in this project: Your friend uses it to embed queries and answers into ChromaDB. You also use it in arize_logger.py to compute the real confidence score by measuring cosine similarity between query, answer, and retrieved context.

The same model is used for both storage and evaluation on purpose — if confidence is measured with a different model than storage uses, the scores would be meaningless. Consistency is critical.
🔍
RAG — Retrieval Augmented Generation
Core Architecture Pattern · Not a Tool, a Concept

What RAG is: A design pattern where instead of asking the LLM a question cold, you first retrieve relevant documents from a knowledge base and inject them into the prompt as context. The LLM then answers based on that context rather than just its training data.

Why it matters: LLMs have a fixed knowledge cutoff. They can't know about your company's internal documents, last week's research paper, or anything that happened after training. RAG solves this by providing fresh, specific context at query time.

The flow in this project: Question → embed question → search ChromaDB for top-2 similar chunks → build prompt: "Here is context: [chunks]. Now answer: [question]" → Phi-3 answers using that context → much better answer than asking Phi-3 cold.

RAG is why "Not Useful" feedback works: when a user says an answer was bad, the improved answer gets stored in ChromaDB. Next time someone asks a similar question, the improved answer is retrieved as context — the system literally learned from its mistake.
📡
Arize AI
Observability · ML Monitoring

What it is: A cloud platform specifically designed to monitor machine learning models in production. Unlike Grafana (which monitors infrastructure), Arize monitors the quality of AI outputs — drift, hallucinations, confidence distribution.

What it does here: Every single /chat call logs: the query, the answer, the confidence score, latency, and whether it was low-confidence. Arize builds a 3D embedding map showing how different queries cluster — which reveals patterns of where the AI fails.

Standard monitoring (Prometheus/Grafana) tells you CPU usage. Arize tells you "40% of questions about Topic X have confidence below 0.5" — which is what you actually need to detect knowledge gaps. This is the difference between infrastructure monitoring and ML observability.
📐
Confidence Score
Your Core Innovation · Cosine Similarity

What it is: A float between 0 and 1 that measures how well the LLM's answer is grounded in the retrieved context. It is computed using cosine similarity — the angle between two embedding vectors.

Formula:

score = avg( cos_sim(query_emb, context_emb), cos_sim(answer_emb, context_emb) )

Interpretation: Score near 1 = answer is strongly grounded in retrieved context. Score near 0 = answer has nothing to do with the context — the model was guessing. Score = 0.30 exactly = no context was retrieved at all (hardcoded floor).

This replaced your friend's random.uniform(0.3, 0.9) — a placeholder that made the metric meaningless. Real confidence based on semantic distance is what makes the self-healing loop actually intelligent.
📊
Weights & Biases (W&B)
Experiment Tracking · Scientific Proof

What it is: A platform that logs ML experiments — metrics over time, model configs, hyperparameters — and lets you compare runs visually.

What it does here: Every /chat call logs: confidence score, latency, source (rag vs model), and whether repair was triggered. After a repair pipeline runs, it logs a Table showing before/after confidence improvement.

What the panel sees: A line graph of confidence scores going up after each repair run. A table showing "Before repair: 0.31, After repair: 0.80". This is your scientific proof that the system improves.

W&B is used in industry at OpenAI, Hugging Face, and DeepMind to track experiments. Showing it in your project proves you understand production ML workflows, not just toy projects.
📈
Prometheus + Grafana
Infrastructure Metrics · Live Dashboard

Prometheus: A time-series metrics database. It scrapes /metrics from FastAPI every 15 seconds and stores values like request count, response time, confidence score, and the update_trigger_flag.

Grafana: A visualization layer that queries Prometheus and renders live graphs. Your dashboard shows all metrics in one view — request rate, confidence over time, repair trigger events, KB version.

Key metric — update_trigger_flag: When this gauge = 1, the Prefect sensor detects it and fires the repair pipeline. This is the bridge between monitoring and action.

Prometheus + Grafana is the industry standard for application monitoring, used at Uber, GitLab, and Netflix. Your friend already included it. You extended it with ML-specific gauges (confidence_score, kb_version).
🔄
Prefect
Orchestrator · Workflow Automation

What it is: A Python-native workflow orchestration tool. You define pipelines as @flow functions with @task steps. Prefect handles retry logic, scheduling, logging, and the UI.

What it does here: The ingest_pipeline @flow is the repair pipeline — it loads logs, cleans data, scans for PDFs, runs Ray processing, and updates Delta Lake. The sensor loop polls every 60 seconds and triggers this flow when confidence drops.

The UI (localhost:4200) shows every pipeline run as a graph — nodes light up green as steps complete. This is your "orchestration proof" for the panel.

Your friend included Prefect but left the flow as a stub (it did nothing). You upgraded it to a real pipeline that actually processes data. This is the key technical contribution — turning a placeholder into a working system.
⚡
Ray
Distributed Processing · Parallelism

What it is: A Python framework for distributed and parallel computing. With @ray.remote, any function can be executed in parallel across all CPU cores.

What it does here: When a PDF is ingested, it's split into ~300-word chunks. Instead of embedding each chunk one at a time (slow, sequential), Ray runs embed_chunk() on all chunks simultaneously across all your CPU cores.

Visual proof (localhost:8265): The Ray dashboard shows CPU utilisation spiking across all cores during processing. Processing 100 chunks takes seconds instead of minutes.

A normal for-loop embedding would take 2 minutes for a 50-page PDF. Ray does it in 8 seconds. The dashboard showing all cores at 100% is one of the most visually impressive moments in your demo.
📦
Delta Lake
Data Versioning · Time-Travel Storage

What it is: An open-source storage layer that adds versioning, ACID transactions, and rollback to file-based data storage. Think of it as "Git for your data files."

What it does here: Every time the repair pipeline runs and adds new document chunks to ChromaDB, Delta Lake records it as a new version — with timestamp, operation type, and list of chunk IDs. Version 0 is the initial state. Version 1 is after first repair. And so on.

Rollback capability: If a corrupt PDF was accidentally ingested, you can roll back to the previous version in one command. No data is ever permanently lost.

Delta Lake proves to the panel that your system is enterprise-grade. Any production data system must have versioning and rollback. Most student projects skip this — including it immediately elevates your project to a different tier.
☁️
Pulumi
Infrastructure as Code · IaC

What it is: A tool that lets you define and manage infrastructure (servers, containers, databases, networks) using regular Python code instead of clicking through dashboards or writing YAML.

What it does here: The infra/__main__.py file defines all Docker containers — ChromaDB, FastAPI, Prefect, Prometheus, Grafana — as Python objects. Running pulumi up creates all of them. Running pulumi destroy removes all of them.

Your friend's placeholder: The original infra/__main__.py just exported the project name — it did nothing. You replaced it with real infrastructure code that provisions the entire system.

Infrastructure as Code means your entire deployment is reproducible. Any developer can clone the repo, run pulumi up, and have the identical environment. This is how real engineering teams work.
🔁
ArgoCD
GitOps · Continuous Deployment

What it is: A Kubernetes-native tool that watches your GitHub repository and automatically deploys any changes to your cluster. This is called "GitOps" — Git is the single source of truth for what should be running.

What it does here: ArgoCD watches your repo. When you push a code change, ArgoCD detects it within 3 minutes and applies the change to the Kubernetes cluster automatically — no kubectl apply, no Jenkins job, no human action required.

Self-healing infra: If someone manually changes something in the cluster (configuration drift), ArgoCD detects the mismatch and reverts it back to what Git says it should be.

ArgoCD closes the loop at the infrastructure level — just as Prefect closes the loop at the data level. Together, they mean the entire system manages itself: code, deployment, and data pipelines.

Section 05

The Theory Behind the Architecture

Every design choice in this project maps to a real computer science or ML engineering concept. Here's the theory behind each one.

🔁 Closed-Loop Autonomous Learning Control Systems Theory

The entire system is based on closed-loop control — a concept from control systems engineering. In an open loop, you give input and hope the output is correct (a standard chatbot). In a closed loop, the output is measured, compared to a desired target, and the difference (error) drives a corrective action.

Here: desired output = high confidence answer (≥0.65). Actual output = measured confidence. Error = gap below threshold. Corrective action = ingest new documents. The loop closes when confidence rises above the threshold. This is identical in principle to a thermostat, a PID controller, or any feedback control system.

📐 Cosine Similarity as Confidence Linear Algebra · Vector Space Models

Embeddings exist in a high-dimensional vector space (384 dimensions here). In this space, semantically similar texts are close together — their vectors point in similar directions. Cosine similarity measures the angle between two vectors: 1.0 = identical direction (perfectly similar), 0.0 = perpendicular (unrelated), -1.0 = opposite meaning.

Confidence is the average of two cosine similarities: how similar the query is to the retrieved context (did we find relevant info?) and how similar the answer is to the context (did the model use what we gave it?). A high score means both questions are answered with yes — the model gave a grounded, relevant response.

This is grounded in the distributional hypothesis in linguistics: words and texts that occur in similar contexts have similar meanings. This is what all transformer models are trained on.

🗃️ Vector Databases and Approximate Nearest Neighbour Search Information Retrieval · ANN Algorithms

ChromaDB doesn't do an exact brute-force search through all stored vectors (that would be O(n) and slow at scale). It uses Approximate Nearest Neighbour (ANN) algorithms — specifically HNSW (Hierarchical Navigable Small Worlds) — which organise vectors in a graph structure that allows O(log n) similarity search.

The distance threshold of 0.5 in your friend's code is a semantic relevance gate: only vectors within cosine distance 0.5 are considered "relevant enough to use as context." Vectors further away are probably about different topics and would confuse the LLM rather than help it.

⚡ Parallel Computing with Ray Distributed Systems · Task Parallelism

Embedding each text chunk requires running it through a neural network (SentenceTransformer). Each call is independent — the result of embedding chunk 1 doesn't affect chunk 2. This is called embarrassingly parallel — problems that can be split into completely independent sub-tasks with no communication needed between them.

Ray uses Python's @ray.remote decorator to turn a function into a distributed task. embed_chunk.remote() submits the task to Ray's scheduler, which assigns it to a free CPU core. ray.get(futures) blocks until all tasks complete and collects results. With 8 cores and 100 chunks, theoretical speedup is 8× — and in practice it's closer to 6× due to serialisation overhead.

📦 Data Versioning and ACID Properties Database Theory · Lakehouse Architecture

Delta Lake brings ACID properties to file-based storage. Atomicity: each ingestion either fully completes or is rolled back. Consistency: the version counter is always accurate. Isolation: concurrent writes don't corrupt each other. Durability: once written, data survives crashes.

The _delta_log/ folder is the transaction log — every operation is recorded as a JSON entry before it's applied. This "write-ahead log" pattern is the same technique used by PostgreSQL and MySQL for crash recovery. Delta Lake adds time travel on top: any previous version can be read by specifying its version number, because the log records exactly what changed in each version.

☁️ Infrastructure as Code (IaC) DevOps · Declarative vs Imperative Infra

Traditional infrastructure management is imperative: you run commands (docker run, kubectl apply) and hope the system ends up in the right state. IaC is declarative: you describe the desired state in code, and the tool figures out what commands to run to achieve it.

Pulumi takes this a step further by using real programming languages (Python here) instead of DSLs like Terraform's HCL. This means you can use loops, conditionals, functions, and libraries when defining infrastructure — making it far more powerful for complex systems. The state file Pulumi maintains tracks what it last deployed, so subsequent pulumi up runs only change what actually changed.

🔁 GitOps and Desired State Reconciliation Site Reliability Engineering · Kubernetes Patterns

GitOps is a philosophy where the Git repository is the single source of truth for what should be running. ArgoCD continuously compares the desired state (what's in Git) with the actual state (what's running in Kubernetes). When they differ, ArgoCD reconciles — it makes the actual state match the desired state.

This solves configuration drift: the gradual divergence between what was deployed and what actually runs, caused by manual changes, hotfixes, or ad-hoc patches. With ArgoCD, any manual change is automatically reverted within minutes. The only way to change what runs is to change Git — making every deployment tracked, versioned, and auditable.

📡 ML Observability vs Infrastructure Monitoring MLOps · Production AI Systems

Traditional monitoring (Prometheus/Grafana) answers: "Is the server up? How much CPU? What's the latency?" These are infrastructure metrics. They tell you the system is running, but not whether it's running well — whether the AI is actually giving good answers.

ML observability answers different questions: "What is the distribution of confidence scores? Are certain types of questions failing more than others? Is model performance drifting over time?" Arize AI is built specifically for these questions. Its embedding visualisation clusters queries in 3D space — topics the model knows well form tight, high-confidence clusters, while knowledge gaps appear as loose, low-confidence outlier regions. This is information you cannot get from any infrastructure metric.
